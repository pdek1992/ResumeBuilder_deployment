# Security Audit Report – Resume Builder SaaS

**Date:** 2026‑05‑15

---

## 1. Scope
This audit focuses on the **PDF/Docx download API** (`/api/downloads/pdf` and `/api/downloads/docx`) and the **payment‑gateway integration** that gates those downloads.  The goal is to identify any vectors that could allow a user to bypass the 24‑hour paid‑access check and obtain a resume document without a valid purchase.

---

## 2. Architecture Overview
```
Client (Browser)                ──►  Next.js API Routes (Node.js)
   │                                   │
   │   GET /api/downloads/pdf?token=…  │
   │                                   │   • verifyDownloadToken(token)
   │                                   │   • getSupabaseServerClient() → Supabase Auth
   │                                   │   • getActiveResumePass(user.id)
   │                                   │   • render PDF via @react-pdf/renderer
   │                                   │   • logUserAction()
   ▼                                   ▼
PDF file (binary stream)    ◄───►  Response
```
* **Token** – a short‑lived download token generated by `/api/downloads/token/route.ts` (not shown here).
* **Pass** – a record in the `payments` schema (`resume_pass` table) that stores `user_id`, `resume_id`, `expires_at` (24 h after purchase) and a flag `used`.
* **Payment verification** – Razorpay order creation (`/api/payments/create-order`) and webhook validation (`/api/payments/verify`).

---

## 3. Threat Model & Attack Surfaces
| Surface | Threat | Likelihood | Impact |
|---------|--------|------------|--------|
| **Download token** (URL query) | Token leakage via referer, browser history, logs; replay attack | Medium | Full document download without payment |
| **`getActiveResumePass`** logic | Missing expiry check, race condition, or stale DB cache | Low‑Medium | Unlimited downloads after 24 h |
| **Supabase auth cookie** | Session hijacking, CSRF, XSS stealing cookie | Low | Auth‑bypass to any API route |
| **Razorpay webhook** | Missing signature verification, replay, tampering | Low | Fake “payment success” entries → passes created |
| **Client‑side UI** | Direct `fetch` to API without token (via dev tools) | Low | Same as token leakage |
| **Rate limiting** | Brute‑force enumeration of tokens | Low | DoS / token harvesting |
| **Logging** | Token printed in server logs or error messages | Low | Token exposure to log collectors |

## 4. Detailed Findings
### 4.1. Download Token Handling
* The token is sent as a **query string** (`?token=…`). Browsers include the full URL in the **Referer** header when following external links – any third‑party site could capture the token.
* Tokens are stored in the database only for a short period (implementation not shown). The API does **not set any cache‑control** other than `no-store`, which is good, but the response still contains `content‑disposition: attachment`, allowing the file to be saved locally and later re‑uploaded.
* No **one‑time‑use** enforcement – the same token can be reused until it expires.

### 4.2. `getActiveResumePass` Validation
* The code only checks `if (!pass) return fail("Export access expired", 403);`. It **does not** verify that the `pass` belongs to the **requested resume** (`payload.resumeId`). If a user obtains another user’s pass, they could download any resume they have permission to.
* **Expiry** is enforced by the DB column (`expires_at`). However, there is no **clock‑skew handling**; an attacker could manipulate the system clock on the server (if they have OS access) to extend validity.
* The `pass` record does not have a **used flag** – repeated downloads within the 24 h window are allowed without limit.

### 4.3. Supabase Authentication & CSRF
* Authentication is performed via `supabase.auth.getUser()`. The session cookie is **HttpOnly** but not explicitly set with `SameSite=Strict`. If the cookie were to be sent on a cross‑site request, a CSRF attacker could trigger a download on behalf of the victim (provided they know a valid token).
* No **CSRF token** is required for the GET download endpoint (GET is considered safe, but here it triggers a privileged action).

### 4.4. Payment Gateway Integration (Razorpay)
* Order creation (`/api/payments/create-order`) returns an order ID. The verification webhook (`/api/payments/verify`) should validate the `razorpay_signature` using the secret key. **If the verification step is missing or the secret is hard‑coded in the repo**, an attacker could spoof a successful payment and create a `resume_pass` entry manually.
* No **idempotency key** – duplicate webhook deliveries could create multiple passes.

### 4.5. Client‑Side Exposure
* The **download token** is accessible via the browser’s developer console (`window.location.search`). Users can copy the URL and share it.
* The UI component `resume-preview.tsx` does not obscure the download button; it directly calls the API with the token, meaning any user with DevTools can replay the request.
* No **Content‑Security‑Policy (CSP)** header is set; a malicious script could read the token from the DOM and exfiltrate it.

## 5. Recommendations
### 5.1. Token Hardening
1. **Make tokens one‑time‑use** – store a `used_at` timestamp and reject subsequent uses.
2. **Shorten token lifetime** to a few minutes after generation (e.g., 5 min) and require **re‑generation** after each download request.
3. **Send token in an HTTP header** (e.g., `Authorization: Bearer <jwt>`) instead of query string to avoid Referer leakage.
4. **Sign tokens with HMAC‑SHA256** (use a secret not committed to repo) and include `userId`, `resumeId`, and `exp` claims.

### 5.2. Pass Validation Enhancements
1. Verify that `pass.resume_id === payload.resumeId`; reject if mismatched.
2. Add a **used flag** and enforce a maximum number of downloads per pass (e.g., 1‑3).
3. Use **database‑level TTL** (`expires_at`) and ensure the server clock is NTP‑synced.
4. Log every successful download with IP, User‑Agent, and timestamp for audit.

### 5.3. CSRF & Session Hardening
1. Set Supabase auth cookie with `SameSite=Strict` and `Secure` flags.
2. Require a **CSRF token** for the download endpoint (even though it’s GET, treat it as a state‑changing operation).
3. Enforce **rate limiting** (e.g., 5 download attempts per minute per IP) using a middleware like `express-rate-limit`.

### 5.4. Razorpay Webhook Security
1. Verify `razorpay_signature` using the secret key on every webhook call.
2. Store the **order_id** and **payment_id** in the `payments` table; only create a `resume_pass` after successful verification.
3. Implement **idempotent webhook handling** – check if the payment_id has already been processed.
4. Rotate the webhook secret periodically and keep it out of source control.

### 5.5. Client‑Side Defences
1. Add a **Content‑Security‑Policy** header that disallows inline scripts and restricts script sources.
2. Use **HttpOnly** cookies only; never expose sensitive tokens to the DOM.
3. Obfuscate the download URL in the UI or request it via a **POST** with JSON body containing the token.
4. Enable **Referrer‑Policy: no-referrer** for the download page to stop browsers from sending the full URL as a referer.

### 5.6. Monitoring & Alerts
* Set up alerts for:
  * Unexpected spikes in download requests.
  * Repeated failed token verification.
  * Webhook signature verification failures.
* Store audit logs in a tamper‑evident store (e.g., CloudWatch Logs with immutable retention).

---

## 7. Conclusion
The current implementation provides a solid baseline (Supabase auth, server‑side PDF generation), but **several high‑impact gaps** allow a determined attacker to bypass the 24‑hour payment gate and obtain downloadable resumes. By applying the recommendations above—especially token hardening, strict pass validation, CSRF mitigation, and webhook verification—the platform will achieve **defense‑in‑depth** and meet production‑grade security standards.
